## What Is This System?

This is a **centralized betting coordination platform** that manages multiple mobile devices placing bets across different betting providers (like Evolution Gaming, Pragmatic Play, etc.).

Think of it as a **conductor orchestrating an ensemble**: the central server (backend) decides *what* each device should do and *when*, then sends instructions to all devices. Each device then executes those instructions locally, precisely at the agreed time.

## The Problem It Solves

### Current Challenge
- Managing multiple betting accounts manually is time-consuming and error-prone
- Coordinating bets across many devices requires precise timing
- Distributing betting amounts fairly based on account performance is complex
- Tracking results and managing funds across accounts is difficult
- IP conflicts can trigger provider detection systems

### Our Solution
- **Automated planning**: The system calculates optimal bet distribution automatically
- **Precise coordination**: All devices execute bets at exactly the same moment (±50ms accuracy)
- **Smart allocation**: Betting amounts are distributed based on account commission rates with monotonic constraints
- **Complete tracking**: Every action is logged and auditable
- **IP management**: Automatic conflict detection and proxy rotation
- **Game state awareness**: Devices detect betting windows for higher success rates

## High-Level Flow

### 1. Setup Phase (Admin)
```
Admin creates:
├── Teams (groups of accounts)
├── Accounts (with commission rates)
├── Matchups (Team A vs Team B)
└── Devices (mobile phones with credentials)
```

### 2. Round Preparation (Server)
```
For each round:
├── Check IP conflicts
├── Rotate proxies if needed
├── Run allocation engine
│   ├── Calculate commission-weighted amounts
│   ├── Apply randomization (±15%)
│   └── Enforce monotonic constraints
├── Lock capital
└── Broadcast to devices via WebSocket
```

### 3. Execution Phase (Devices)
```
Each device:
├── Receives round instructions
├── WebView loads betting provider website
├── Waits for betting window to open
├── Executes bet via touch gestures on WebView at precise timestamp
├── Verifies bet placement
└── Reports outcome to server
```

### 4. Settlement Phase (Server)
```
After round completes:
├── Collect results from devices
├── Update capital balances
├── Unlock remaining funds
├── Check auto-pause conditions
└── Generate audit logs
```

## Technology Stack

### Backend
- **Framework**: Laravel 11 (PHP 8.3)
- **Database**: PostgreSQL 16
- **Cache/Queue**: Redis 7
- **WebSocket**: Laravel Reverb
- **Queue Monitoring**: Laravel Horizon
- **Admin Panel**: Filament 3

### Mobile (Android)
- **Language**: Kotlin Native
- **Core**: WebView + AccessibilityService
- **UI Surface**: WebView loads betting provider websites (no native provider apps)
- **Interaction**: AccessibilityService + GestureExecutor for touch injection on WebView (no DOM manipulation or JS injection — indistinguishable from real user)
- **Network**: OkHttp (WebSocket)
- **DI**: Hilt
- **Architecture**: MVVM + Clean Architecture

### DevOps
- **Container**: Docker + Docker Compose
- **Monitoring**: Prometheus + Grafana
- **CI/CD**: GitHub Actions
- **Reverse Proxy**: Nginx

## System Limitations

### Known Constraints
1. **Android Only** - iOS not supported in Phase 1 (AccessibilityService limitation)
2. **Provider Coverage** - Evolution Gaming only in Phase 1, other providers in future phases
3. **Network Dependency** - Requires stable internet for real-time coordination
4. **Time Accuracy** - Limited by network latency (±50ms typical)
5. **OEM Restrictions** - Some Android manufacturers aggressively kill background services
6. **WebView Dependency** - Betting providers must have a functional web interface; provider UI changes may require adapter updates

### Workarounds
- Multiple devices provide redundancy
- Comprehensive OEM-specific guides
- Fallback to time-based execution
- Regular time sync minimizes drift
- Admin monitoring catches issues quickly
